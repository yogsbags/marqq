# Phase 7 Research — Swarm Layer

## Goals & Requirement Mapping
- **SWARM-01**: Priya is the Competitive Intelligence conductor. The run should use the `ConcurrentWorkflow` pattern so it fans out to 10 per-competitor `Watchdog` subtasks (concurrent fetching) and then fans back in for synthesis/insight generation.
- **SWARM-02**: Each Watchdog must perform strictly differential analysis — read `mkg.json` → `competitors[n].last_checked` → only fetch/process content that arrived after that timestamp, so delta-driven work is the norm.
- **SWARM-03**: Corpus-level analysis is handled by `SpreadsheetSwarm`: up to 300 YouTube transcripts and 400 tweets per cycle, cheap filtering via the `Haiku` skill, synthesis via `Sonnet`, with separate queues for filtering vs analysis to control token spend.
- **SWARM-04**: Outcomes land in Supabase `competitive_intelligence` table keyed by `(company_id, competitor_name, week_of)` so downstream reporting and hooks can pick the latest weekly snapshot.
- **SWARM-05**: Priya writes competitor move records/signals into `agent_signals` (e.g., `competitor_content_pub` or `competitor_move`) so `HooksEngine` can dispatch follow-on agents (Priya → Riya) within the same evaluation cycle.

## Architecture Overview
Priya is a supervisor that orchestrates a hybrid, multi-agent swarm:

1. **ConcurrentWorkflow fan-out/in**: The primary swarm runner (`swarm-runner.ts`) uses `Promise.all` (or a scheduler-friendly queue) to spawn 10 watchers simultaneously. Each watcher is responsible for a single competitor, scoped via `mkg.competitors`. After watchers finish, the runner aggregates the delta findings into a single synthesis payload that fuels the final `AgentRunOutput` and `competitive_intelligence` record.
2. **Watchdog per competitor**: Each Watchdog follows the Delta Watch pattern—pull `last_checked`, call connectors (YouTube, Twitter, RSS, press releases, etc.), and skip any content already seen. If no new content is detected, the Watchdog short-circuits without invoking expensive LLM calls, updates `last_checked` to the existing timestamp, and records a zero-token health entry. When new content exists, the Watchdog writes raw snippets (IDs, timestamps, URLs, metadata) into a staging table/queue before handing them to `SpreadsheetSwarm`.
3. **SpreadsheetSwarm pipeline**: Corpus ingestion works in two phases:
   - **Haiku filter** runs on the incoming transcript/tweet batch (max 300/400). This lightweight model drops noise and assigns a relevance score. Filtered rows stream into the analysis phase.
   - **Sonnet synthesis** consumes filtered rows (possibly batched again) to produce structured insights, competitor tactics, and sentiment signals. Sonnet’s output merges with Watchdog metadata before writing to `competitive_intelligence`.
4. **Signal & Hooks integration**: After synthesis, Priya emits one or more `agent_signals` rows (signal type `competitor_move`, `competitor_content_pub`, etc.). `HooksEngine` evaluates these signals on the next heartbeat, cross-references MKG baseline diff, and dispatches Priya again plus Riya (Content Creation) per the Hooks config.
5. **Data & state persistence**: `mkg.competitors[n].last_checked` and `mkg.competitors[n].history` provide checkpoints; `competitive_intelligence` stores summaries keyed by `week_of` so weekly KPIs can be plotted. Watchdog completion metadata (duration, tokens used, delta range) can be appended to a `swarm_watchdog_runs` helper table for debugging/validation.

## Key Data Flows
1. **Run kickoff**: Cron (or Hook-triggered) job fires nightly. Priya loads MKG to gather the 10 competitors, their connectors, and the `last_checked` delta anchors. The cron schedule should live next to `hooks.json` and `agent_signals` to keep triggers coordinated with the existing Hooks System.
2. **Watchdog fetch**: Each Watchdog queries the relevant connectors (YouTube API, Twitter timelines, RSS, etc.) with `published_after=last_checked`. Retrieved resources are appended to a temporary `span` table or `spreadsheet_swarm_batches` collection with metadata (source, timestamp, company_id, competitor_name).
3. **SpreadsheetSwarm filtering**: The `Haiku` filtering agent reads the batch, prunes duplicates, rates relevance, and annotates transcripts/tweets. This stage is intentionally cheap (smaller LLM or retrieval-based heuristics) to keep 300/400 inputs manageable. Filtered items are flagged for Sonnet.
4. **SpreadsheetSwarm synthesis**: The `Sonnet` agent receives the filtered corpus (bounded size), generates structured insights (themes, tactics, metric signals), and hands results back to Priya. This summary includes: new product launches, pricing changes, creative angles, and CTA modifications.
5. **Persistence**: Priya writes a `competitive_intelligence` row with `company_id`, `competitor_name`, `week_of`, `summary`, `themes`, `actions_detected`, `confidence`, `source_run_id`, plus `haiku_filtered_count`/`sonnet_token_usage`. The `agent_signals` table receives a `competitor_move`/`competitor_content_pub` signal referencing the same `run_id`.
6. **Hooks dispatch**: On the next heartbeat, `HooksEngine` reads the new signal(s), validates diff-from-baseline via MKG (e.g., `mkg.competitors[n].baseline_content`) and schedules Riya (Content) along with Priya for follow-up emails or updates. The same hook evaluation also logs `signal_processed_at` to avoid duplicates.

## Required Tables / Collections
- **Supabase `competitive_intelligence`** (SWARM-04): Primary storage for weekly summaries. Columns: `company_id`, `competitor_name`, `week_of`, `summary`, `themes`, `actions`, `confidence`, `haiku_count`, `sonnet_tokens`, `run_id`, `created_at`. Composite PK ensures deduped week-of entries.
- **`agent_signals`** (existing): Priya writes `competitor_move` / `competitor_content_pub` signals here; `HooksEngine` consumes them. Add metadata columns such as `source_run_id`, `competitor_name`, and `signal_payload`.
- **`mkg.competitors` section**: `last_checked`, `baseline_content`, `preferred_connectors`, `watchdog_status`, `watchdog_history` arrays live inside the MKG JSON. Each Watchdog updates `last_checked` (timestamp) and appends computed delta context.
- **`swarm_watchdog_runs` (or similar helper table)**: Stores per-run telemetry for debugging/validation (run_id, competitor_name, delta_start, delta_end, tokens_used, status). Useful to sample watchers and compare to `last_checked`.
- **`spreadsheet_swarm_batches` collection or table**: Optional staging area for transcripts/tweets that captures `batch_id`, `source_type`, `item_count`, `filtered_count`, `unfiltered_payload`, `company_id`, `competitor_name`, `run_id`. Enables replay/testing without re-fetching connectors.
- **`agent_tasks` (existing)**: Each Watchdog can optionally emit follow-ups (e.g., manual review). `Priya` may create tasks when anomalies exceed thresholds.

## Priority Unknowns
- **Connector availability & rate limits**: Which upstream connectors (YouTube API quotas, official Twitter API, RSS, etc.) are available for the 10 competitors, and how to gracefully back off when 429s hit? Need guardrails to avoid failing the entire swarm when one connector dies.
- **Corpus ingestion quotas**: How to enforce 300 transcripts / 400 tweets per cycle within `SpreadsheetSwarm` while keeping `Haiku` / `Sonnet` token budgets under control? Need to confirm where transcripts/tweets live (local storage vs connectors) and whether we can pre-fetch metadata to avoid duplicate downloads.
- **Haiku/Sonnet routing**: The instructions mention cheap filtering and analysis models; need to decide which exact models are used (Groq `compound-mini` for filtering, `llama-3.3-70b` for synthesis?) and how to failover if one is slow. Also, how are filtering results persisted so Sonnet can retry without re-fetching.
- **Concurrency/backpressure**: Running 10 Watchdogs + SpreadsheetSwarm simultaneously could exhaust rate limits or tokens; require tuning for worker concurrency, queue length, and instrumentation to avoid running out of handles in the Node process.
- **Validation hooks**: How to sample nightly runs for validation without interfering with production? Need a deterministic `test_run` or instrumented run to guarantee `Validation Architecture` (below) can access reproducible logs.
- **Signal de-dupe**: `HooksEngine` must not re-fire the same `competitor_move` signal; need to define TTL/dedup keys (maybe `run_id` + `signal_type + competitor`) and persist `signal_processed_at`.

## Validation Architecture
- **Nightly sampling**: Each execution of the swarm job writes a `swarm_validation` entry (or can reuse `swarm_watchdog_runs`) with: timestamps, `run_id`, `company_id`, `competitor_name`, `delta_start/end`, items processed, signals emitted, and whether `HooksEngine` consumed the signal. A lightweight `validation_runner` (could be part of the same cron or a separate API) picks 3 watchers per night (rotate through all 10 weekly) and replays the `delta` using stored metadata to ensure `last_checked` logic is precise.
- **Delta verification**: For sampled competitors, compare the `last_checked` timestamp in MKG before and after the run. Confirm only new items are processed by checking the stored `delta_start`/`delta_end` archive or `spreadsheet_swarm_batches` IDs. Track `new_content_count` vs `haiku_filtered_count`. If a competitor had zero new items, the validation runner asserts there were zero Sonnet calls and no tokens consumed.
- **Signal chain verification**: After the swarm writes `agent_signals`, the validation job queries Supabase to confirm that HooksEngine consumed the signal and inserted a `hook_dispatches` log (add this table if needed). Validation also ensures that Priya → Riya runs occur within 60 seconds per requirement; this can be sampled by watching the `agent_tasks` timeline or `hooks.log`.
- **Weekly synthesis check**: Pick one `competitive_intelligence` row from the current week, replay the associated `spreadsheet_swarm_batches`, and run a dry `Sonnet` invocation (can use stub) to ensure the summary is reproducible. Store the hash of the Sonnet prompt/output in `swarm_validation` so future runs can detect drift or missing context.
- **Alerting**: If validation detects any mismatch (delta processed vs `last_checked`, missing signal, or no `competitive_intelligence` row for a company with new data), raise an entry in `agent_tasks` (priority: high) or escalate via existing monitoring channels. Because every validation run shares the same `run_id`, debugging is easier.

With this architecture, Phase 7 can meet the swarm-specific requirements while providing observable, repeatable verification for nightly execution.
