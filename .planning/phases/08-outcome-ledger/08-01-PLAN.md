## Task 1: Create the outcome ledger migration

1. `database/migrations/outcome-ledger.sql` defines `outcome_ledger` with columns `run_id`, `company_id`, `agent`, `metric`, `baseline_value`, `predicted_value`, `actual_value`, `variance_pct`, `verified_at`, `created_at`.
2. Enforce service-role-only access via RLS policy (same pattern as `agent_run_outputs`).
3. Add indexes on `(company_id, verified_at)` and `(agent, verified_at)` for API lookups.

## Task 2: Implement outcome processor helpers

1. Add `platform/content-engine/outcome-processor.js` exporting `computeVariance({baseline, predicted, actual})` and `writeOutcomeLedgerRow({runId,...})`.
2. Helpers should read KPIs from Supabase `company_kpi_daily` (via existing Supabase client) and call MKG/Memory updates when `variance_pct > 30`.
3. Expose calibration note creation (append to `memory/MEMORY.md`) but leave actual scheduler call to next plan.

## Task 3: Build Arjun verification job

1. Create `platform/crewai/outcome-verifier.py` that loads `hooks.json` (or new config) and queries Supabase for the latest AgentRunOutput rows with an `outcome_prediction`.
2. For each run, resolve actual KPIs from `company_kpi_daily` (matching metric/period) and call `writeOutcomeLedgerRow`.
3. If `variance_pct > 30`, enqueue a memory calibration write (via Supabase or direct `memory/MEMORY.md` append) and optionally raise a calibration signal for Phase 7 hooks.
4. This script is schedulable from Phase 5's scheduler once `hooks.json` includes a weekly entry (hook id `arjun-weekly-outcome`).
